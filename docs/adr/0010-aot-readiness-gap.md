# ADR-0010: Native AOT ‚Äî Cli + App publish-aot is required, not aspirational

**Status**: Accepted (Phase 2 complete: `<PublishAot>` is a required CI gate for every project)
**Date**: 2026-05-11 (Phase 1 ‚Äî readiness gap; amended 2026-05-12 ‚Äî Phase 2A config retirement; amended 2026-05-12 ‚Äî Phase 2B WinAppSDK retirement, AOT now real; amended 2026-05-19 ‚Äî reconciliation pass: composition-root suppression deleted along with Tomlyn, Platform.Windows IL2050/IL2057 site count corrected, CI publish-aot gate confirmed green via workflow_dispatch + artifact upload)

## Context

Native AOT is the right destination for an overlay binary: instant startup, smaller footprint, no JIT. Reality on .NET 10 + WinAppSDK 2.0:

- WinAppSDK 2.0 has improving but uneven AOT support; XAML markup and CsWinRT projection regressions still surface.
- Tomlyn is reflection-based; AOT compatibility needs verification at integration time.
- `System.CommandLine` 2.x has source-generator support and is largely AOT-friendly.

Treating AOT as a v0.1 release blocker is unrealistic. Treating it as "we'll get to it" silently breaks the moment someone introduces reflection.

## Decision

- **Code is AOT-ready from Day 1.** No `dynamic`, no `Activator.CreateInstance(Type)`, no `System.Reflection.Emit`.
- **Analyzers run on every project**, independent of `IsAotCompatible`. `Directory.Build.props` sets `EnableTrimAnalyzer`, `EnableSingleFileAnalyzer`, `EnableAotAnalyzer` to `true` directly. The MSBuild evaluation order leaves these values intact even when a csproj overrides `IsAotCompatible=false` (verified 2026-05-12 via `dotnet msbuild -getProperty`). After Phase 2 every publishable assembly ‚Äî Cli, App, Platform.Windows, Storage, XTask ‚Äî sets `IsAotCompatible=true`, so no project currently exercises the fallback; the analyzer flow is the load-bearing guarantee that new reflection trips a diagnostic regardless of where it lands.
- **Diagnostics are explicit errors via `.editorconfig`.** A per-diagnostic `dotnet_diagnostic.IL2026.severity = error` (and the rest of the IL2xxx / IL3xxx series) block lives next to the existing CA / IDE severity overrides. The blanket `TreatWarningsAsErrors=true` would have promoted them anyway; the explicit block documents intent so a future contributor silencing one of these reads "AOT readiness, ADR-0010" rather than treating the failure as background noise.
- **`<PublishAot>` is real for every project.** Phase 2A retired Tomlyn (ADR-0015 ‚Äî config file gone). Phase 2B retired WinAppSDK + CsWinRT projection (ADR-0009 amendment ‚Äî dcomp-direct via CsWin32). Both Cli and App are `IsAotCompatible=true`; `just publish-aot` (Cli `win-x64`) and `just publish-dist-aot` (App `win-x64`) run as required CI gates. Three narrower canaries cover the leaf shapes: `aot-xtask` (linux-x64, ~4.5 MB native binary, runs `strict-code` end-to-end), `aot-storage` (library AOT-publish, linux-x64, ~900 KB `.so`), and `aot-canary` / `aot-canary-dist` (Cli + App, win-x64, windows-latest). All four are required.
- **JSON via STJ source generators only.** `[JsonSerializable(typeof(...))]` on a partial `JsonSerializerContext`. Today the runtime path writes JSON via `Utf8JsonWriter` directly (no reflection), so no context exists yet ‚Äî `BannedSymbols.txt` already rejects the reflection-based `JsonSerializer.Serialize/Deserialize` overloads, so the first consumer that needs serialization will be forced into the source-generated path.
- **TOML loader retired (Phase 2A, ADR-0015).** The Tomlyn-backed `Linerule.Config` project and its `ConfigLoader` are gone ‚Äî tunables now live on `UserConfig.Default` as compile-time constants. The Tomlyn-incompatible JSON overload entries in `BannedSymbols.txt` are retained as a general AOT-incompat tripwire even though their original consumer is removed.
- **`LoggerRoot` exposes sink paths through a typed interface.** `IPathfulSink` (in `Linerule.Platform.Windows.Diagnostics`) replaces the previous `sink.GetType().GetProperty("Path")` reflection. The old form would have silently returned `null` after any property rename anyway, so the AOT fix also closed a latent bug.
- **Composition-root suppression ‚Äî none required.** Phase 2A retired the Tomlyn-based config-load phase entirely; `BootDag.Default` and its constituent `Phase` arrows carry no `[RequiresUnreferencedCode]` / `[RequiresDynamicCode]` annotations, so neither `Linerule.Cli` nor `Linerule.App` needs an `AotBoundary` static-class wrapper. The planned `AotTestBoundary` shim in the now-retired `Linerule.Config.Tests` project is gone too. New reflection introduced anywhere in the runtime tree still trips `IL2026` / `IL3050` ‚Üí build error via `Directory.Build.props` (`EnableTrimAnalyzer`, `EnableSingleFileAnalyzer`, `EnableAotAnalyzer`, all `true`).
- **`Linerule.XTask strict-code` is the regex lower bound.** Fourteen rules grouped by category: code hygiene (5), AOT/trim reflection (8), SQL safety (1). The AOT rules cover `dynamic`, `Reflection.Emit`, `Activator.CreateInstance(typeof)`, `MakeGenericType`, `MakeGenericMethod`, `Type.GetType("‚Ä¶")`, `Assembly.Load/LoadFrom/LoadFile`, and the `.GetType().GetX(` / `typeof(Foo).GetX(` member-lookup patterns. `*.g.cs`, `obj/`, `bin/`, and `Generated/` are excluded so source-generator output doesn't self-trip. Tested at `tests/Linerule.XTask.Tests/` ‚Äî one positive case per rule plus structural cases (clean source, empty tree, excluded paths, tests/ sweep).
- **`BannedSymbols.txt` is the symbol-level upper bound.** Adds Tomlyn-incompatible JSON overloads, `Reflection.Emit` builders, `Activator.CreateInstance(Type)`, `Type.MakeGenericType` / `MethodInfo.MakeGenericMethod`, every `Assembly.Load*` overload, and `Type.GetProperty/GetField(string)`. `Type.GetType(string)` and `Type.GetMethod(string)` are deliberately *not* in `BannedSymbols.txt` because `Microsoft.Windows.CsWin32` emits them in `WinRTCustomMarshaler.g.cs`; those two are covered by the strict-code regex (hand-written `.cs` only) plus the IL2057 analyzer (silenced only in `Linerule.Platform.Windows` where the generator output lives ‚Äî everywhere else IL2057 stays an error).
- **`Linerule.Platform.Windows` silences IL2050 / IL2057.** The csproj `<NoWarn>$(NoWarn);IL2050;IL2057</NoWarn>` is the documented escape hatch for CsWin32-generated COM marshalling diagnostics. Four hand-written sites additionally carry per-method `[UnconditionalSuppressMessage]` annotations as documentation: `D3D11Devices` (device-creation factory), `OverlayWindow` (window-class registration), and two D2D/DWrite factory calls in `Hud/HudRenderer`. These per-method annotations re-engage as the primary signal if the CsWin32 generator-output footprint ever shrinks enough to drop the csproj `NoWarn`.

## Consequences

- `just publish-aot` and `just publish-dist-aot` are green required CI gates on windows-latest as of Phase 2B; verified end-to-end via `workflow_dispatch` + `actions/upload-artifact` (`aot-cli-win-x64` / `aot-app-win-x64`). The AOT build path is exercised on every PR ‚Äî regressions surface immediately rather than waiting for a release-time port. The original "release event, not a refactor" framing was retrospective: Phase 2A/2B turned the goal post into a tripwire.
- A contributor who introduces reflection trips three independent gates: the strict-code regex (`just lint`), the BannedSymbols analyzer (`RS0030` at build time), and the trim/AOT Roslyn analyzers (`IL2xxx` / `IL3xxx` at build time). All three gates fail the build by default; silencing any of them requires either an explicit `[UnconditionalSuppressMessage]` with a Justification (the ADR-0006 escape hatch) or an ADR amendment.
- After Phase 2A/2B the remaining reflection lives entirely in `Linerule.Platform.Windows`: CsWin32 COM marshalling (`IL2050`) and the WinRT marshaller's `Type.GetType` (`IL2057`), silenced via the project `<NoWarn>` plus the four per-method `[UnconditionalSuppressMessage]` annotations described above. Each suppression is documented and scoped; removing any of them (e.g. when CsWin32 fully migrates to function pointers) collapses the corresponding scope without touching unrelated paths.
- Test coverage for strict-code lives in `tests/Linerule.XTask.Tests/`; every rule has a positive fixture under `Path.GetTempPath()` so the fixtures do not themselves get scanned by the CI sweep. New rules require both a positive and a negative case before merge.
