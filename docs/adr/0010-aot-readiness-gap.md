# ADR-0010: Native AOT — Cli + App publish-aot is required, not aspirational

**Status**: Accepted (Phase 2 complete: `<PublishAot>` is a required CI gate for every project)
**Date**: 2026-05-11 (Phase 1 — readiness gap; amended 2026-05-12 — Phase 2A config retirement; amended 2026-05-12 — Phase 2B WinAppSDK retirement, AOT now real)

## Context

Native AOT is the right destination for an overlay binary: instant startup, smaller footprint, no JIT. Reality on .NET 10 + WinAppSDK 2.0:

- WinAppSDK 2.0 has improving but uneven AOT support; XAML markup and CsWinRT projection regressions still surface.
- Tomlyn is reflection-based; AOT compatibility needs verification at integration time.
- `System.CommandLine` 2.x has source-generator support and is largely AOT-friendly.

Treating AOT as a v0.1 release blocker is unrealistic. Treating it as "we'll get to it" silently breaks the moment someone introduces reflection.

## Decision

- **Code is AOT-ready from Day 1.** No `dynamic`, no `Activator.CreateInstance(Type)`, no `System.Reflection.Emit`.
- **Analyzers run on every project**, independent of `IsAotCompatible`. `Directory.Build.props` sets `EnableTrimAnalyzer`, `EnableSingleFileAnalyzer`, `EnableAotAnalyzer` to `true` directly. The MSBuild evaluation order leaves these values intact even when a csproj overrides `IsAotCompatible=false` (verified 2026-05-12 via `dotnet msbuild -getProperty`), so the `Linerule.Cli` / `Linerule.App` / `Linerule.Platform.Windows` overrides — kept because Tomlyn and CsWin32 / WinAppSDK genuinely block AOT publish — do not silence the diagnostic flow on those projects.
- **Diagnostics are explicit errors via `.editorconfig`.** A per-diagnostic `dotnet_diagnostic.IL2026.severity = error` (and the rest of the IL2xxx / IL3xxx series) block lives next to the existing CA / IDE severity overrides. The blanket `TreatWarningsAsErrors=true` would have promoted them anyway; the explicit block documents intent so a future contributor silencing one of these reads "AOT readiness, ADR-0010" rather than treating the failure as background noise.
- **`<PublishAot>` is real for every project.** Phase 2A retired Tomlyn (ADR-0015 — config file gone). Phase 2B retired WinAppSDK + CsWinRT projection (ADR-0009 amendment — dcomp-direct via CsWin32). Both Cli and App are `IsAotCompatible=true`; `just publish-aot` (Cli `win-x64`) and `just publish-dist-aot` (App `win-x64`) run as required CI gates. Three narrower canaries cover the leaf shapes: `aot-xtask` (linux-x64, ~4.5 MB native binary, runs `strict-code` end-to-end), `aot-storage` (library AOT-publish, linux-x64, ~900 KB `.so`), and `aot-canary` / `aot-canary-dist` (Cli + App, win-x64, windows-latest). All four are required.
- **JSON via STJ source generators only.** `[JsonSerializable(typeof(...))]` on a partial `JsonSerializerContext`. Today the runtime path writes JSON via `Utf8JsonWriter` directly (no reflection), so no context exists yet — `BannedSymbols.txt` already rejects the reflection-based `JsonSerializer.Serialize/Deserialize` overloads, so the first consumer that needs serialization will be forced into the source-generated path.
- **TOML via Tomlyn (DOM mode).** `TomlSerializer.Deserialize<TomlTable>` returns a typed DOM; `RawConfigDeserializer` walks it manually. Both call sites carry `[RequiresUnreferencedCode]`/`[RequiresDynamicCode]` so the trim/AOT chain propagates to the composition root. If the chain ever loses an annotation, the analyzer trips at the next caller — see "Composition-root suppression" below. Should the small schema mapper ever need to disappear, the hand-rolled replacement is straightforward.
- **`LoggerRoot` exposes sink paths through a typed interface.** `IPathfulSink` (in `Linerule.Platform.Windows.Diagnostics`) replaces the previous `sink.GetType().GetProperty("Path")` reflection. The old form would have silently returned `null` after any property rename anyway, so the AOT fix also closed a latent bug.
- **Composition-root suppression.** The annotated chain (`BootDag.Default`, `ConfigLoader.Load`, `ConfigLoader.ParseString`) surfaces at exactly three places: `Linerule.Cli.AotBoundary`, `Linerule.App.AotBoundary`, and `Linerule.Config.Tests.AotTestBoundary`. Each is a single static class with `[UnconditionalSuppressMessage(... Justification = "ADR-0010 ...")]` on the wrapping methods; call sites invoke the wrapper instead of the annotated member directly. New reflection introduced anywhere else still trips `IL2026` / `IL3050` → build error, because the suppression is scoped to exactly the documented gap.
- **`Linerule.XTask strict-code` is the regex lower bound.** Fourteen rules grouped by category: code hygiene (5), AOT/trim reflection (8), SQL safety (1). The AOT rules cover `dynamic`, `Reflection.Emit`, `Activator.CreateInstance(typeof)`, `MakeGenericType`, `MakeGenericMethod`, `Type.GetType("…")`, `Assembly.Load/LoadFrom/LoadFile`, and the `.GetType().GetX(` / `typeof(Foo).GetX(` member-lookup patterns. `*.g.cs`, `obj/`, `bin/`, and `Generated/` are excluded so source-generator output doesn't self-trip. Tested at `tests/Linerule.XTask.Tests/` — one positive case per rule plus structural cases (clean source, empty tree, excluded paths, tests/ sweep).
- **`BannedSymbols.txt` is the symbol-level upper bound.** Adds Tomlyn-incompatible JSON overloads, `Reflection.Emit` builders, `Activator.CreateInstance(Type)`, `Type.MakeGenericType` / `MethodInfo.MakeGenericMethod`, every `Assembly.Load*` overload, and `Type.GetProperty/GetField(string)`. `Type.GetType(string)` and `Type.GetMethod(string)` are deliberately *not* in `BannedSymbols.txt` because `Microsoft.Windows.CsWin32` emits them in `WinRTCustomMarshaler.g.cs`; those two are covered by the strict-code regex (hand-written `.cs` only) plus the IL2057 analyzer (silenced only in `Linerule.Platform.Windows` where the generator output lives — everywhere else IL2057 stays an error).
- **`Linerule.Platform.Windows` silences IL2050 / IL2057.** The csproj `<NoWarn>$(NoWarn);IL2050;IL2057</NoWarn>` is the documented escape hatch for CsWin32-generated COM marshalling diagnostics. The 3 hand-written D3D11 / D2D / DWrite factory call sites in `HudRenderer` additionally carry per-method `[UnconditionalSuppressMessage]` annotations as documentation; they would re-engage if Platform.Windows ever becomes AOT-compatible and the csproj NoWarn is removed.

## Consequences

- A future `just publish-aot` becoming green is a release event, not a refactor — the code never has to be ported. The known blockers are upstream (Tomlyn reflection, WinAppSDK 2.0 AOT regressions, CsWin32 COM marshalling); each can be addressed in isolation without touching application code.
- A contributor who introduces reflection trips three independent gates: the strict-code regex (`just lint`), the BannedSymbols analyzer (`RS0030` at build time), and the trim/AOT Roslyn analyzers (`IL2xxx` / `IL3xxx` at build time). All three gates fail the build by default; silencing any of them requires either an explicit `[UnconditionalSuppressMessage]` with a Justification (the ADR-0006 escape hatch) or an ADR amendment.
- Existing reflection — Tomlyn in `Linerule.Config`, CsWin32 COM marshalling in `Linerule.Platform.Windows`, the WinRT marshaller's `Type.GetType` — is each scoped, annotated, and suppressed at exactly one documented point. Removing any one of those suppression points (e.g. replacing Tomlyn with a hand-rolled mapper) collapses the corresponding scope without touching unrelated paths.
- Test coverage for strict-code lives in `tests/Linerule.XTask.Tests/`; every rule has a positive fixture under `Path.GetTempPath()` so the fixtures do not themselves get scanned by the CI sweep. New rules require both a positive and a negative case before merge.
